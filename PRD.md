## 0. 구현 완료 요약 (Implementation Summary, 2026-07-22)

Phase 1~4를 별도 프로토타입이 아니라 **하나의 코드베이스**(`output/가계부/index.html`)로 이어 붙여 전부 구현했다. 브라우저에서 실제 조작(거래 CRUD, 예산, 통계, 백업/복원, 카테고리·규칙 관리)과, 합성 영수증 이미지로 **실제 Tesseract.js OCR 인식 전 과정**(사진 선택 → 텍스트 인식 → 날짜/금액/상호 추출 → 카테고리 자동 매핑 → 검수 화면 프리필 → 저장)을 검증 완료했다.

- **OCR 엔진**: Tesseract.js v5, `https://cdn.jsdelivr.net/npm/tesseract.js@5/dist/tesseract.min.js`에서 **최초 사용 시점에 지연 로딩**(lazy-load)으로 CDN에서 스크립트를 가져온다. 6번 섹션 Open Question이었던 "CDN vs 폴더 동봉" 중 **CDN 방식으로 확정** — 파일 용량을 가볍게 유지하는 대신, 영수증 인식 기능은 최초 사용 시 인터넷 연결이 필요하다(기본 가계부 기능은 오프라인에서도 100% 동작).
- **사진 입력**: `<input type="file" accept="image/*" capture="environment">` — 모바일 브라우저에서 탭하면 후면 카메라가 바로 열리며, 별도 앱 설치 없이 "휴대폰으로 사진 찍어서 바로 입력" 요구사항을 그대로 충족한다.
- **항목 추출 방식**: ML이 아닌 **정규식 기반 휴리스틱**으로 구현 — 금액은 "합계금액/총액/결제금액" 등 키워드 인접 숫자 우선, 없으면 텍스트 내 최대 숫자를 대체 사용. 날짜는 `YYYY.MM.DD` 계열 패턴 매칭. 상호명은 사업자번호·전화번호·주소 등 잡음 라인을 제외한 첫 유효 라인을 채택.
- **카테고리 자동 분류**: 상호명/원문에 포함된 키워드 사전(기본 61개, 스타벅스·이마트·CGV·약국 등 국내 주요 브랜드/업종 커버) 매칭. 설정 화면에서 사용자가 규칙을 추가·삭제할 수 있고, 이 규칙은 백업 JSON에 함께 저장되어 다른 기기로 이전 가능.
- **검수 플로우**: 인식 결과는 바로 저장되지 않고, Phase 1의 거래 입력 폼에 프리필된 채로 "영수증 인식 결과" 모달이 뜬다 — 안내 배너, 원문 텍스트 보기 토글, 모든 필드 수정 가능, 저장 시 일반 수동 입력과 동일한 코드 경로로 처리된다.
- **실패/저품질 처리**: 인식된 텍스트가 4자 미만이면 "다시 촬영해주세요" 토스트를 띄우되, 모달은 그대로 열어 사용자가 수동으로 채울 수 있게 한다.
- **개인정보**: 이미지는 브라우저 내에서만 처리되고 어떤 서버로도 전송되지 않는다(Tesseract.js는 완전 클라이언트 사이드).

## 1. Problem Statement

사용자는 집에서 가족 단위로 사용할 가계부가 필요하다. 기존에 잘 알려진 가계부 앱(뱅크샐러드, 토스, Toshl, Money Manager 등)이 제공하는 수준의 기본기(수입/지출 기록, 카테고리 분류, 예산 관리, 통계)는 갖추되, 매번 영수증을 보고 손으로 항목·금액·카테고리를 입력해야 하는 번거로움이 가장 큰 마찰 지점이다. 영수증 사진 한 장으로 글씨를 인식하고 항목을 자동 분류해 입력까지 끝내주는 기능이 있으면 기록 누락과 입력 피로를 크게 줄일 수 있다.

## 2. Goals

1. 별도 서버·회원가입 없이 브라우저에서 바로 쓸 수 있는 단일 파일(또는 폴더) 가계부를 만든다 (본 저장소 산출물 규칙 준수).
2. 수입/지출 기록, 카테고리 분류, 예산 설정, 월별·카테고리별 통계 등 기존 가계부 앱과 동등한 기본 사용성을 Phase 1에서 확보한다.
3. 영수증 사진 촬영/업로드만으로 날짜·금액·상호명을 인식해 지출 항목을 자동 생성한다 (Phase 2~3).
4. 인식된 상호명을 바탕으로 카테고리(식비/카페/마트/교통 등)를 자동으로 매핑해 수동 분류 작업을 최소화한다 (Phase 3).
5. OCR 인식은 무료 수단(클라이언트 사이드 오픈소스 라이브러리 또는 무료 티어 API)만 사용해 운영 비용이 들지 않도록 한다.

## 3. Non-Goals

- **다중 기기 동기화/클라우드 백업**: 서버·DB 인프라가 필요해 "빌드 없는 단일 산출물" 원칙과 충돌 — v1은 브라우저 로컬 저장(localStorage/IndexedDB) 한 기기 기준으로 한정. 필요 시 향후 별도 인프라 프로젝트로 분리(Future Consideration).
- **은행/카드사 자동 연동(오픈뱅킹, 스크래핑)**: 인증·보안 요건이 크고 이 프로젝트 범위를 벗어남 — 수동 입력과 영수증 사진 인식으로 대체.
- **다인 동시 편집/가족 공유 계정**: 계정 시스템이 전제되어야 하므로 v1 범위 밖. 한 기기를 가족이 함께 쓰는 방식으로 대체 가능.
- **영수증 원본 이미지의 서버 저장/외부 전송**: 개인 재정 정보가 담긴 민감 이미지이므로, 클라우드 API를 쓰더라도 이미지를 영구 보관하는 서버는 두지 않는다(인식 후 즉시 폐기).
- **완벽한 OCR 정확도 보장**: 무료 OCR의 한계(구겨진 영수증, 저해상도, 감열지 흐림 등)로 100% 인식은 불가능 — 대신 "인식 후 사용자가 확인/수정하는 검수 단계"로 보완(Non-goal이 아니라 설계 전제).

## 4. User Stories

- **As a 가계부 사용자**, 나는 지출/수입 항목을 날짜·금액·카테고리·메모와 함께 수동으로 입력할 수 있어서, 원하는 어떤 거래든 기록할 수 있다.
- **As a 가계부 사용자**, 나는 카테고리별(식비, 교통, 주거, 의료 등)로 지출을 분류해서, 어디에 돈을 많이 쓰는지 한눈에 볼 수 있다.
- **As a 가계부 사용자**, 나는 카테고리별 월 예산을 설정해두고 현재 사용액과 잔여액을 비교해서, 과소비를 미리 인지할 수 있다.
- **As a 가계부 사용자**, 나는 월별/카테고리별 지출 통계(막대·원형 차트)를 볼 수 있어서, 소비 패턴을 파악할 수 있다.
- **As a 가계부 사용자**, 나는 영수증 사진을 찍거나 갤러리에서 선택하면, 앱이 날짜·금액·상호명을 자동으로 읽어와서, 직접 타이핑하지 않아도 된다.
- **As a 가계부 사용자**, 나는 인식된 상호명(예: "스타벅스", "이마트")을 바탕으로 카테고리가 자동으로 지정된 상태의 입력 화면을 받아서, 확인 후 저장 버튼만 누르면 되게 하고 싶다.
- **As a 가계부 사용자**, 나는 OCR이 잘못 읽었거나 카테고리를 잘못 추정했을 때, 저장 전에 값을 손쉽게 수정할 수 있어서, 최종 데이터의 정확성을 유지할 수 있다.
- **As a 가계부 사용자**, 나는 자동 입력 시 사용한 상호명→카테고리 매핑 규칙을 직접 추가/수정할 수 있어서, 내가 자주 가는 가게에 맞게 분류 정확도를 계속 높일 수 있다.
- **As a 모바일 사용자**, 나는 스마트폰 브라우저에서 카메라로 영수증을 바로 촬영해 인식 과정을 진행할 수 있어서, 별도 앱 설치 없이 그 자리에서 기록을 끝낼 수 있다.

## 5. Phased Roadmap (단계별 구현 계획)

큰 기능(기본 가계부 + OCR 자동 분류)을 한 번에 만들지 않고, 아래 순서로 단계별 지시·검수를 전제로 진행한다.

### Phase 1 — 기본 가계부 (수동 입력 기반)

기존 가계부 앱과 동등한 최소 기능. OCR 없이도 완전히 동작하는 독립적인 v1.

| # | 요구사항 | 우선순위 | 인수 기준 |
|---|---|---|---|
| 1-1 | 거래 추가/수정/삭제 (수입·지출 구분, 날짜, 금액, 카테고리, 상호/메모) | P0 | Given 입력 폼, When 필수값(날짜·금액·유형) 입력 후 저장, Then 목록에 즉시 반영된다 |
| 1-2 | 기본 카테고리 프리셋 제공 (일반적인 가계부 앱 수준, 지출 12종 + 수입 4종) + 사용자 정의 카테고리 추가 | P0 | 프리셋 내장, "+카테고리 추가"로 이름/아이콘 지정 가능 |
| 1-3 | 거래 내역 목록(일별 그룹핑, 최신순) | P0 | 날짜별로 묶여 표시되고 각 항목에 금액·카테고리·메모가 보인다 |
| 1-4 | 월별 요약(총수입/총지출/순잔액) | P0 | 상단 요약 카드에 이번 달 수입·지출·잔액이 표시된다 |
| 1-5 | 카테고리별 예산 설정 및 사용률 표시 | P0 | 카테고리에 월 예산 입력 시, 현재 사용액 대비 진행률 바(%)로 표시, 초과 시 경고 색상 |
| 1-6 | 월별/카테고리별 통계 (도넛·막대 차트) | P0 | 이번 달 카테고리별 지출 비중을 차트로 확인 가능, 월 전환(이전/다음 달) 가능 |
| 1-7 | 데이터 로컬 저장(브라우저 재실행/새로고침에도 유지) | P0 | localStorage 또는 IndexedDB에 저장, 새로고침 후에도 데이터 유지 |
| 1-8 | 데이터 내보내기/가져오기 (JSON 백업 파일을 PC로 다운로드, 동일 파일로 복원) | **P0(확정)** | "백업 다운로드" 클릭 시 `.json` 파일이 PC에 저장되고, "복원"으로 동일 파일을 선택하면 데이터가 그대로 복원된다 |
| 1-9 | 반응형 레이아웃(모바일 우선) | P0 | 360px 폭에서도 입력·조회가 불편 없이 동작 |
| 1-10 | 단일 산출물(빌드 불필요, CSS/JS 인라인) | P0 | `output/가계부/`에 자체완결형 HTML(+필요 시 보조 JS 파일)로 배포 |

### Phase 2 — 영수증 사진 입력 + 텍스트 인식(OCR)

Phase 1 위에 "사진 → 텍스트" 파이프라인만 추가. 아직 자동 분류·자동 입력까지는 하지 않고, 인식된 원문 텍스트를 사용자에게 보여주는 단계.

| # | 요구사항 | 우선순위 | 인수 기준 |
|---|---|---|---|
| 2-1 | 영수증 사진 촬영/업로드 UI (모바일 카메라 직접 호출 + 파일 선택) | P0 | `<input type="file" accept="image/*" capture="environment">` 기반, 모바일에서 탭 시 카메라가 바로 열린다 |
| 2-2 | 클라이언트 사이드 OCR 처리 (Tesseract.js, 한국어 언어팩) | P0 | 이미지 업로드 후 별도 서버 호출 없이 브라우저 내에서 텍스트가 추출된다 |
| 2-3 | 인식 진행 상태 표시(로딩 인디케이터) | P0 | OCR 처리 중 스피너/진행률이 표시되고, 완료 시 결과 화면으로 전환 |
| 2-4 | 인식 원문 텍스트 미리보기 | P0 | 사용자가 OCR 원문 전체를 확인할 수 있는 화면 제공(디버깅/검증 목적) |
| 2-5 | 인식 실패/저품질 이미지 안내 | P1 | 텍스트가 거의 추출되지 않은 경우 "다시 촬영해주세요" 안내 표시 |

### Phase 3 — 항목 자동 추출 및 카테고리 자동 분류

OCR 원문에서 실제 가계부 항목(날짜/금액/상호명)을 파싱하고, 상호명 기반으로 카테고리를 추정한다.

| # | 요구사항 | 우선순위 | 인수 기준 |
|---|---|---|---|
| 3-1 | 금액 추출 규칙 (합계/총액 키워드 우선 매칭: "합계", "총액", "받을금액" 등 + 숫자 패턴) | P0 | 영수증 내 여러 숫자 중 최종 결제 금액을 최우선 규칙으로 선택 |
| 3-2 | 날짜 추출 규칙 (YYYY-MM-DD, YY.MM.DD, MM/DD 등 정규식 패턴 매칭) | P0 | 인식 텍스트에서 날짜 형식을 찾아 거래 날짜 필드에 자동 채움, 없으면 오늘 날짜로 기본값 |
| 3-3 | 상호명 추출 규칙 (영수증 상단 첫 줄 또는 "가맹점명" 라벨 기준) | P0 | 상호명 후보를 추출해 메모/상호 필드에 채움 |
| 3-4 | 상호명 → 카테고리 자동 매핑 (키워드 사전 기반: "스타벅스/카페"→카페, "이마트/마트"→마트·장보기 등) | P0 | 내장 매핑 사전에 있는 상호명이면 해당 카테고리가 자동 선택된 상태로 입력 폼이 뜬다 |
| 3-5 | 매핑 사전 사용자 편집(상호명-카테고리 규칙 추가/수정) | P1 | 설정 화면에서 규칙을 추가하면 이후 인식 시 즉시 반영된다 |
| 3-6 | 매핑되지 않은 상호명은 "기타" 카테고리로 기본 지정 | P0 | 사전에 없는 상호명은 카테고리 미지정 대신 "기타"로 채워져 저장 누락을 방지 |

### Phase 4 — 검수 UI 및 저장 플로우 완성

인식·분류 결과를 사용자가 최종 확인/수정 후 한 번에 저장하는 화면으로 Phase 1~3을 통합한다.

## 6. OCR/영수증 인식 기술 옵션 검토

무료로 사용 가능한 선택지를 비교했다. **v1 기본 채택안은 Tesseract.js(클라이언트 사이드, 완전 무료·오프라인)** 이다.

| 옵션 | 무료 여부 | 실행 위치 | 장점 | 단점 | 채택 여부 |
|---|---|---|---|---|---|
| **Tesseract.js** | 완전 무료(오픈소스, API 키 불필요) | 브라우저 내(클라이언트) | 서버·API 키 불필요 → "빌드 없는 단일 HTML" 원칙에 완벽히 부합, 이미지가 외부로 나가지 않아 프라이버시 안전 | 한글 인식 정확도가 상용 API 대비 낮음, 저해상도/구겨진 영수증에서 오인식 잦음, 초기 언어팩 로딩(수 MB) 필요 | **Phase 2 기본 채택** |
| Google ML Kit (온디바이스) | 무료 | 네이티브 앱(Android/iOS) 전용 | 정확도 높고 오프라인 지원 | 웹 브라우저에서 사용 불가(네이티브 SDK) — 이번 프로젝트가 웹 산출물인 이상 채택 불가 | 제외 (v1) — 추후 네이티브 앱 전환 시 재검토(P2) |
| Naver Clova OCR | 무료 티어 존재(월 일정 건수) | 서버 API 호출 | 한글 영수증 인식 정확도 우수 | API 키 필요(클라이언트에 노출 위험), CORS로 인해 별도 프록시 서버 필요 → "서버 없음" 원칙과 충돌, 무료 한도 초과 시 과금 | 제외(v1), 정확도 문제 시 P1 대안으로 재검토 |
| Google Cloud Vision API | 월 최초 1,000건 무료 | 서버 API 호출 | 다국어·정확도 우수 | 키 관리·과금 설정 필요, 서버 프록시 필요 | 제외(v1), 동일 사유 |
| OCR.space | 무료 API 키 티어 제공 | 서버 API 호출(브라우저에서 직접 호출 가능) | 키 발급이 간단하고 브라우저에서 직접 호출 가능(CORS 허용) | 무료 티어 요청 수 제한, 키가 클라이언트 코드에 노출됨(개인용 로컬 사용 전제라면 허용 가능한 리스크) | **P1 대안**: Tesseract.js 정확도가 부족할 경우 사용자 본인 명의 무료 키를 발급받아 교체 투입 |

**권장 방향**: Phase 2~3은 Tesseract.js로 구현해 "무설정·무비용·완전 로컬"을 유지한다. 실제 사용 후 한글 영수증 인식률이 기대에 못 미치면, 사용자가 직접 발급한 OCR.space 무료 키를 옵션으로 입력해 대체 엔진으로 전환할 수 있는 구조(설정 화면에 "OCR 엔진 선택" 토글)를 P1로 열어둔다.

## 7. Success Metrics

개인/가정용 산출물이므로 정량 트래킹 대신 **완료 기준(Definition of Done)** 으로 성공을 판단한다:

- **Phase 1 완료 기준**: 수동으로 거래 10건 이상 입력 후 월별 요약·카테고리 통계·예산 진행률이 오류 없이 계산된다.
- **Phase 2 완료 기준**: 실제 영수증 사진 5장(마트/카페/식당/편의점/약국 등 종류별)으로 테스트 시, 최소 3장 이상에서 사람이 읽을 수 있는 수준의 텍스트가 추출된다.
- **Phase 3 완료 기준**: 위 5장 중 최소 3장에서 날짜·금액·상호명 중 2개 이상 필드가 올바르게 자동 채워진다.
- **Phase 4 완료 기준**: 검수 화면에서 잘못된 필드를 수정해 저장하면, Phase 1 목록/통계에 정확히 반영된다(수동 입력과 결과 동일).
- **개인정보 보호 기준**: 영수증 이미지가 어떤 외부 서버로도 전송되지 않음을 네트워크 탭으로 확인.

## 10. Timeline Considerations

- 별도 마감 기한 없음(개인/가정용 산출물).
- 사용자가 단계별로 지시할 예정이므로, 본 PRD는 Phase 1~4 순서로 진행하되 각 Phase 착수 전 세부 지시(디자인 톤, 카테고리 목록 확정 등)를 반영해 확정한다.
